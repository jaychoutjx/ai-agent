## 二、后端选型

### 2.1 Python vs Java vs Go vs Node

| 候选 | 选 / 不选 | 原因 |
|------|----------|------|
| **Python** | ✅ 选 | LangChain / LlamaIndex 生态唯一选择；AI 生态全在这里 |
| Java | ❌ | 有 Spring AI 但生态远落后；调试 Pythonic 库困难 |
| Go | ❌ | 无成熟 LangChain；NLP 库稀缺 |
| Node.js | ❌（候选） | LangChain.js 落后于 Python 半年到一年 |

**结论**：AI 应用首选 Python。

### 2.2 Web 框架：FastAPI

| 候选 | 选 / 不选 | 原因 |
|------|----------|------|
| **FastAPI** | ✅ | async-first / Pydantic 强类型 / 自动 OpenAPI / 性能与 Node 相当 |
| Flask | ❌ | 同步为主；扩展靠各种社区插件 |
| Django | ❌ | 太重；ORM 强但我们要的是 API 网关 |
| Tornado | ❌ | 异步但生态小 |

**为什么是 FastAPI：**
- 流式响应（SSE）原生支持
- 依赖注入清晰
- Pydantic v2 校验性能优秀（Rust 实现）
- 自动 `/docs` Swagger UI 提升联调效率

### 2.3 LLM 编排：LangChain + LangGraph

| 候选 | 选 / 不选 | 原因 |
|------|----------|------|
| **LangChain** | ✅ | 链式调用最成熟；LCEL 优雅；社区生态最大 |
| **LangGraph** | ✅ | LangChain 官方出品的状态图，Agent 必备 |
| LlamaIndex | ❌（部分借鉴）| RAG 强但 Agent 弱 |
| 自研 | ❌ | 重复造轮子；招人时也要求懂 LangChain |
| Semantic Kernel | ❌ | 微软系，生态偏 .NET |

**LangChain 的取舍：**

✅ 优点
- 抽象统一（Runnable 接口贯穿所有组件）
- 集成几乎所有 LLM / 向量库 / 工具
- LCEL 用 `|` 拼链路，支持流式 / 批处理 / 异步零成本切换

⚠️ 缺点（已知雷）
- 版本变化大（0.1 / 0.2 / 0.3 API 不兼容） → 锁版本到 `>=0.3.7`
- 部分二级集成质量参差 → 我们绕过 `langchain-milvus` 直接用原生 `MilvusClient`
- 文档不够稳 → 关键路径自己写测试覆盖

### 2.4 依赖管理：uv

| 候选 | 选 / 不选 | 原因 |
|------|----------|------|
| **uv** | ✅ | Rust 实现，比 pip 快 10-100x；锁文件原生；社区接受度上升 |
| pip + venv | ❌ | 慢；锁文件要靠 pip-tools |
| poetry | ❌（次选）| 慢；社区开始向 uv 迁 |
| pipenv | ❌ | 不维护了 |

**实战体验**：`uv sync` 装齐 70 个依赖只用 30 秒，pip 同样的活要 5 分钟。

### 2.5 日志：loguru

| 候选 | 选 / 不选 | 原因 |
|------|----------|------|
| **loguru** | ✅ | 零配置；颜色 / 文件轮转 / 异常美化全套 |
| logging（标准库）| ❌ | 配置繁琐 |
| structlog | ❌（次选）| 结构化更强但学习成本高，简单项目用不到 |

**踩过的坑**：容器内非 root 用户写不了 `logs/` 目录 → 改造为 `_try_create_log_dir()`，失败时只走 stdout（12-Factor 原则）。

## 三、向量库选型：Milvus

### 3.1 候选对比

| 库 | 部署 | 性能 | 生态 | 选 / 不选 |
|----|------|------|------|----------|
| **Milvus** | 自托管 / Zilliz Cloud | 千万级仍稳 | LangChain / LlamaIndex 一等公民 | ✅ |
| Pinecone | SaaS | 不错 | 一等公民 | ❌（要钱+境外）|
| Chroma | embedded / 服务端 | 百万级以下 | 一等公民 | ❌（小数据玩具）|
| Qdrant | 自托管 / SaaS | 强 | 一等公民 | ⚪（备选）|
| Weaviate | 自托管 / SaaS | 强 | 一等公民 | ⚪（备选）|
| pgvector | PostgreSQL 扩展 | 一般 | 一等公民 | ❌（中小型可用）|
| Faiss | 内存库 | 快但无服务 | 工具库 | ❌（不能持久化）|

### 3.2 为什么 Milvus

- **企业级**：千万级向量稳定运行；分布式可扩展
- **国内合规**：可完全本地化部署
- **索引丰富**：HNSW / IVF / DiskANN / GPU
- **Schema 支持**：可在向量旁存元数据 + 标量字段过滤

### 3.3 索引选择：HNSW vs IVF

| 维度 | HNSW | IVF_FLAT |
|------|------|----------|
| 召回率 | 高（90%+）| 中 |
| 延迟 | 低 | 中 |
| 内存 | 大（~3x 向量本身）| 小 |
| 构建速度 | 慢 | 快 |
| 适用场景 | < 千万级，追求精度 | 千万级以上，省内存 |

**我们的选择：HNSW**，参数 `M=16, efConstruction=200, ef=64`。
原因：知识库通常几万到百万级，对延迟和召回敏感，内存还撑得住。

### 3.4 距离度量：COSINE

`text-embedding-v3` 默认归一化，COSINE = 1 - 内积，语义相似度更直观。

### 3.5 为什么不用 langchain-milvus

实测发现 `langchain-milvus` 的封装：
- 自动建表逻辑过于黑盒
- 不易控制 schema
- partition 切换有 bug

→ 直接用 `pymilvus.MilvusClient` 自己封一层，schema / 索引 / 加载完全可控。

## 四、LLM 选型：阿里云百炼

### 4.1 为什么选 Qwen

| 候选 | 选 / 不选 | 原因 |
|------|----------|------|
| **百炼 Qwen** | ✅ | 国内合规；中文好；价格便宜；一站式（chat / embed / rerank）|
| OpenAI | ❌ | 需国外通道；合规风险 |
| DeepSeek | ⚪（次选）| 性能强但多模态弱；与 Qwen 切换成本几乎为零 |
| 智谱 GLM | ⚪ | 性能不错 |
| Claude | ❌ | 同 OpenAI |

### 4.2 模型矩阵

| 任务 | 模型 | 特点 | 备注 |
|------|------|------|------|
| 通用对话 | `qwen-plus` | 平衡性价比 | 默认主力 |
| 复杂推理 | `qwq-plus` | reasoning 强 | Agent 备用 |
| Embedding | `text-embedding-v3` | 1024 维 | **每批最多 10 条** ← 注意点 |
| Rerank | `gte-rerank-v2` | 中文友好 | Cross-Encoder |

### 4.3 走 OpenAI 兼容协议

DashScope 提供 OpenAI 兼容协议（`/compatible-mode/v1`），直接用 `langchain-openai` 接入：

```python
ChatOpenAI(
    model=settings.qwen_chat_model,
    base_url=settings.dashscope_base_url,  # https://dashscope.aliyuncs.com/compatible-mode/v1
    api_key=settings.dashscope_api_key,
)
```

**好处**：换模型只改 `model` 参数，代码零改动。

## 五、前端选型

### 5.1 Next.js 15 + React 19

| 候选 | 选 / 不选 | 原因 |
|------|----------|------|
| **Next.js 15** | ✅ | App Router / SSR / standalone 输出便于 Docker |
| Vite + React | ⚪ | 简单但缺 SSR；适合纯 SPA |
| Remix | ❌ | 生态小于 Next |
| Nuxt（Vue） | ❌ | 团队 React 栈 |

**为什么 Next.js**：
- App Router 文件即路由，组件即可流式
- `output: "standalone"` 让 Docker 镜像 ~150 MB
- React 19 的 `use()` Hook 对流式数据友好

### 5.2 状态管理：Zustand

```typescript
// Zustand 全部代码
const useChatStore = create<ChatStore>((set) => ({
  messages: [],
  appendMessage: (msg) => set((s) => ({ messages: [...s.messages, msg] })),
  ...
}));
```

| 候选 | 选 / 不选 | 体积 |
|------|----------|------|
| **Zustand** | ✅ | ~1 KB |
| Redux Toolkit | ❌ | 大；boilerplate 多 |
| Jotai | ⚪ | 原子化思路好但学习成本 |
| React Context | ❌ | 频繁更新有重渲染问题 |

### 5.3 SSE 客户端：fetch-event-source

| 候选 | 选 / 不选 | 原因 |
|------|----------|------|
| **@microsoft/fetch-event-source** | ✅ | 支持 POST + 自定义 header；支持 AbortController |
| 原生 EventSource | ❌ | 只支持 GET，不能传 header |
| axios | ❌ | 不原生支持 SSE |

**关键场景**：发送对话需要 POST body 带 history → 必须能 POST。

### 5.4 样式：Tailwind CSS

- 原子化、按需打包，bundle 小
- 配合 `clsx` + `tailwind-merge` 写 utility 函数
- 不用搞 CSS 命名规范

### 5.5 包管理：pnpm

- 比 npm/yarn 快，磁盘占用小（hard link）
- 严格模式避免幻影依赖
- corepack 锁版本，Docker 中靠 `"packageManager": "pnpm@11.1.2"` 字段稳定

**踩过的坑**：pnpm 11 要求 Node 22+，最初 Dockerfile 用 `node:20-alpine` 构建失败 → 切到 `node:22-alpine`。

## 六、容器化：Docker + Compose

### 6.1 为什么 Docker

- 环境一致：`docker compose up` 即跑
- Milvus 等中间件自动起来
- CI/CD 友好

### 6.2 多阶段构建

| 阶段 | 作用 |
|------|------|
| `deps` | 装依赖（pnpm / uv），缓存层 |
| `builder` | 构建产物（next build） |
| `runner` | 仅复制产物 + 运行 |

后端 Python 镜像：~400 MB（基础 slim + 依赖）
前端 Next.js standalone：~150 MB

### 6.3 为什么不直接 Kubernetes

- MVP 阶段单机够用
- 学习成本太高
- 设计上保留了水平扩展的可能（无状态后端、外部存储）

## 七、检索算法选型

### 7.1 召回算法

| 算法 | 角色 | 强项 | 弱项 |
|------|------|------|------|
| **向量检索** | 主路 | 语义匹配 | 专有名词失效 |
| **BM25** | 辅路 | 关键词 / 缩写命中 | 不懂语义 |

→ 双路并行 + RRF 融合，互补。

### 7.2 融合算法：RRF

```
score(d) = Σ 1 / (k + rank_i(d))
```

为什么 RRF 不用加权和：
- 向量分数 0~1，BM25 分数 0~100+，量纲不同
- 归一化方式（min-max / softmax）选择困难
- RRF 只用排名，天然消除尺度问题，论文证明效果好

### 7.3 精排：Cross-Encoder Reranker

```
向量 / BM25  →  Bi-Encoder（独立编码 query 和 doc）
                 速度快，精度有限

Reranker     →  Cross-Encoder（query 和 doc 拼一起编码）
                 看到对方，精度高，但慢
```

业界共识：粗排 Top-50 + 精排 Top-5。
论文实验：单纯向量 Recall@5 ≈ 70%，加 Reranker 能 → 90%+。

## 八、技术选型决策表（一图流）

| 我们选了 | 因为 | 代价 | 缓解 |
|---------|------|------|------|
| Python | AI 生态唯一 | 性能比 Go 慢 | async-first |
| FastAPI | 异步 + 类型 | 学习曲线 | 团队大多熟 |
| LangChain | 生态最大 | 版本变化大 | 锁版本 + 关键路径自测 |
| Milvus | 企业级向量库 | 部署复杂 | Docker Compose |
| 百炼 Qwen | 国内合规 + 便宜 | 不如 GPT-4 | 关键场景换 qwq-plus |
| HNSW | 召回 + 延迟优 | 内存大 | 数据量级匹配 |
| Next.js 15 | SSR + standalone | 复杂 | App Router 上手快 |
| Zustand | 极简 | 大型项目可能不够 | 当前规模够用 |
| Docker | 一致性 | 资源占用 | 多阶段构建 |
