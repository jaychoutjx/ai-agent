# 07 · 部署运维

> 从 0 到生产。Docker Compose 一键启动 + 生产化要做哪些事。

## 一、本地开发部署（最快路径）

### 1.1 前置要求

| 工具 | 版本 | 用途 |
|------|------|------|
| Docker Desktop | ≥ 24 | 容器运行时 |
| Docker Compose | v2 | 编排 |

> Windows 用户：建议用 Docker Desktop + WSL2，性能比 Hyper-V 好。

### 1.2 5 分钟跑起来

```bash
# 1. 配置 .env
cp .env.example .env
# 编辑 .env，至少填：DASHSCOPE_API_KEY=sk-xxx

# 2. 一键启动
docker compose up -d

# 3. 看日志（可选）
docker compose logs -f backend frontend

# 4. 访问
# http://localhost:3300       前端
# http://localhost:8800/docs  Swagger UI
# http://localhost:9001       MinIO 控制台（minioadmin/minioadmin）
```

### 1.3 端口映射

| 端口 | 服务 | 用途 |
|------|------|------|
| 3300 | frontend | Next.js 前端 |
| 8800 | backend | FastAPI |
| 19530 | milvus | gRPC |
| 9091 | milvus | HTTP / metrics |
| 9001 | minio | 控制台 |

## 二、Docker Compose 架构

```
                         aikb-net  (bridge)
                              │
         ┌────────────────────┼────────────────────┐
         │                    │                    │
    ┌────────┐          ┌──────────┐         ┌──────────┐
    │ etcd   │          │  minio   │         │  milvus  │
    │ (元数据)│         │ (对象存储)│         │  主节点   │
    └────┬───┘          └────┬─────┘         └─────┬────┘
         │                   │                     │
         └───────────────────┴─────────────────────┘
                              ↑
                      depends_on:
                      condition: service_healthy
                              │
                  ┌───────────┴───────────┐
                  │                       │
            ┌──────────┐            ┌──────────┐
            │ backend  │ ←(依赖)──  │ frontend │
            │ FastAPI  │            │ Next.js  │
            └──────────┘            └──────────┘
                 ↑                       ↑
            8800（host）            3300（host）
```

### 2.1 服务清单

| 服务 | 镜像 | 角色 |
|------|------|------|
| etcd | quay.io/coreos/etcd:v3.5.18 | Milvus 元数据 |
| minio | minio/minio:RELEASE.2024-05 | Milvus 对象存储 |
| milvus | milvusdb/milvus:v2.5.14 | 向量数据库 |
| backend | 本地构建（Python 3.11）| API 服务 |
| frontend | 本地构建（Node 22）| Web UI |

### 2.2 网络模型

- 所有容器在 `aikb-net` 桥接网络
- 容器间通过**服务名**互访（`milvus:19530`，不是 `localhost`）
- 仅 `backend` / `frontend` / `milvus` / `minio` 暴露端口到 host

### 2.3 数据持久化（命名卷）

```yaml
volumes:
  etcd-data:        # Milvus 元数据
  minio-data:       # Milvus 对象数据
  milvus-data:      # Milvus 索引文件
  backend-uploads:  # 上传文件
  backend-data:     # 文档元数据 JSON
```

> `docker compose down` 不会清空，加 `-v` 才会。

### 2.4 健康检查 + 启动顺序

```yaml
backend:
  depends_on:
    milvus:
      condition: service_healthy   # 等 Milvus 健康
```

每个服务都有 `healthcheck`：

```yaml
milvus:
  healthcheck:
    test: ["CMD", "curl", "-f", "http://localhost:9091/healthz"]
    interval: 30s
    start_period: 90s   # 启动期间不算失败（Milvus 慢启动）
    timeout: 20s
    retries: 3
```

避免后端启动时 Milvus 还没 ready，连不上挂掉。

## 三、Dockerfile 详解

### 3.1 后端（多阶段）

```dockerfile
# stage 1: 基础 + 依赖
FROM python:3.11-slim AS base
WORKDIR /app
RUN pip install --no-cache-dir uv -i https://pypi.tuna.tsinghua.edu.cn/simple
COPY pyproject.toml uv.lock* ./
RUN uv sync --frozen --no-cache

# stage 2: 应用代码
COPY app/ ./app/
COPY scripts/ ./scripts/

# 非 root 用户
RUN useradd -m -u 1000 app && chown -R app:app /app
USER app

EXPOSE 8800
HEALTHCHECK --interval=30s --timeout=10s \
  CMD curl -f http://localhost:8800/api/v1/health || exit 1

CMD ["uv", "run", "uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8800"]
```

### 3.2 前端（三阶段 standalone）

```dockerfile
ARG NODE_VERSION=22-alpine

# stage 1: deps
FROM node:${NODE_VERSION} AS deps
WORKDIR /app
RUN corepack enable
COPY package.json pnpm-lock.yaml pnpm-workspace.yaml ./
RUN pnpm install --frozen-lockfile

# stage 2: builder
FROM node:${NODE_VERSION} AS builder
WORKDIR /app
RUN corepack enable
COPY --from=deps /app/node_modules ./node_modules
COPY . .
ARG NEXT_PUBLIC_API_BASE_URL
ENV NEXT_PUBLIC_API_BASE_URL=${NEXT_PUBLIC_API_BASE_URL}
RUN pnpm build

# stage 3: runner（仅 standalone 产物）
FROM node:${NODE_VERSION} AS runner
WORKDIR /app
ENV NODE_ENV=production
RUN addgroup -g 1001 nodejs && adduser -S nextjs -u 1001
COPY --from=builder --chown=nextjs:nodejs /app/.next/standalone ./
COPY --from=builder --chown=nextjs:nodejs /app/.next/static ./.next/static
COPY --from=builder --chown=nextjs:nodejs /app/public ./public
USER nextjs
EXPOSE 3300
ENV PORT=3300 HOSTNAME=0.0.0.0
CMD ["node", "server.js"]
```

**关键点：**
- `output: "standalone"` 让 Next.js 把依赖打进 `.next/standalone`，镜像 ~150 MB
- `corepack` 锁定 pnpm 版本（与 `package.json` 的 `packageManager` 字段联动）
- 非 root 用户 `nextjs:1001`

### 3.3 镜像体积对比

| 镜像 | 体积 |
|------|------|
| 后端（python:3.11-slim + 70 个依赖） | ~400 MB |
| 前端（node:22-alpine + standalone）| ~150 MB |
| Milvus 全家桶（etcd + minio + milvus）| ~1.5 GB |

## 四、环境变量

`.env` 主要字段（详见 `.env.example`）：

```bash
# 必填
DASHSCOPE_API_KEY=sk-xxx                    # 阿里云百炼

# 可选
TAVILY_API_KEY=tvly-xxx                     # 联网搜索（不填则降级 mock）

# 开发期默认值
APP_PORT=8800
MILVUS_HOST=localhost                       # Docker 中会被 compose 覆盖为 milvus
MILVUS_PORT=19530

# 模型选择（一般不改）
QWEN_CHAT_MODEL=qwen-plus
QWEN_EMBEDDING_MODEL=text-embedding-v3
QWEN_RERANKER_MODEL=gte-rerank-v2

# CORS（生产改成自己的域名）
CORS_ORIGINS=http://localhost:3300,http://127.0.0.1:3300
```

### 4.1 关键覆盖（Docker Compose）

```yaml
backend:
  env_file: .env
  environment:
    MILVUS_HOST: milvus       # 容器间用服务名
    APP_HOST: 0.0.0.0          # 监听全部网卡
```

`environment` 优先级 > `env_file`，所以 `.env` 中的 `MILVUS_HOST=localhost` 会被覆盖为 `milvus`。

## 五、常用运维命令

```bash
# 看服务状态
docker compose ps

# 看实时日志
docker compose logs -f backend
docker compose logs -f --tail 200 milvus

# 重启某个服务
docker compose restart backend

# 重新构建（代码改动后）
docker compose up -d --build backend

# 进入容器
docker exec -it aikb-backend bash
docker exec -it aikb-frontend sh

# 停止（保留数据）
docker compose down

# 完全重置（清空所有数据 ⚠️）
docker compose down -v

# 看磁盘占用
docker system df -v
```

## 七、12-Factor App 自检

| 因子 | 我们做到 | 备注 |
|------|---------|------|
| 1. 代码库 | ✅ | 单仓 |
| 2. 依赖 | ✅ | uv lock + pnpm lock |
| 3. 配置 | ✅ | 全走环境变量 |
| 4. 后端服务 | ✅ | Milvus / DB 都是外部资源 |
| 5. 构建/发布/运行 | ✅ | Dockerfile 多阶段 |
| 6. 进程 | ✅ | 无状态（除内存仓库需替换）|
| 7. 端口绑定 | ✅ | 自带 web server |
| 8. 并发 | ⚪ | 单进程；生产开多 worker |
| 9. 易处理 | ✅ | 优雅关闭 |
| 10. 开发与线上等价 | ✅ | Docker 一致 |
| 11. 日志 | ✅ | stdout 优先（容器化原则）|
| 12. 管理进程 | ✅ | scripts/ 下放运维脚本 |

## 八、监控指标建议（落地清单）

| 指标 | 阈值 | 行动 |
|------|------|------|
| 后端 5xx 错误率 | > 1% | 报警 |
| LLM 调用平均延迟 | > 5s | 排查模型/网络 |
| LLM token 日成本 | > 预算 80% | 告警 |
| Milvus 内存 | > 80% | 扩容或拆分 |
| 上传失败率 | > 5% | 排查 OSS |
| Agent 平均迭代次数 | > 4 | 优化 prompt |

## 九、典型故障排查

### Q1. 后端起不来 / `Permission denied: 'logs'`

**原因**：容器内非 root 用户写不了 `logs/` 目录。

**已解决**：`logger.py` 的 `_try_create_log_dir()` 失败时只走 stdout。

如果还有问题，检查容器是否被 mount 了不可写的卷。

### Q2. 前端连不上后端 / CORS 错误

```
Access to fetch at 'http://localhost:8800/...' from origin 'http://localhost:3300'
has been blocked by CORS policy
```

检查 `.env` 里的 `CORS_ORIGINS` 是否包含前端 origin。

### Q3. Milvus `collection not loaded`

```bash
docker exec -it aikb-backend uv run python scripts/reset_milvus.py
```

或：
```python
client.load_collection(COLLECTION_NAME)
```

### Q4. `port 3300 already in use`

主机的 3300 被本地 `pnpm dev` 占了：

```powershell
Get-NetTCPConnection -LocalPort 3300 | Select OwningProcess
Stop-Process -Id <pid>
```

### Q5. 前端构建失败 `ERR_UNKNOWN_BUILTIN_MODULE`

pnpm 11 要求 Node 22+。已在 Dockerfile 锁 `node:22-alpine` + `package.json` 写 `"packageManager": "pnpm@11.1.2"`。

## 十、独立部署 vs Compose 全栈

| 场景 | 推荐 |
|------|------|
| 本地开发 + 自己跑 backend | `docker-compose.infra.yml`（仅基础设施）|
| 给别人演示 | `docker-compose.yml`（一键全栈）|
| 上生产 | 拆开：基础设施用 K8s/托管，业务用单独 Dockerfile |
