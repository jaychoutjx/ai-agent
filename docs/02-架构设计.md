## 二、分层架构

### 2.1 后端分层

```
┌────────────────────────────────────────────────┐
│  api/v1/  (路由层)                              │
│   - 接收 HTTP / SSE                             │
│   - Pydantic 校验                                │
│   - 调用 services 层                             │
└────────────────────┬───────────────────────────┘
                     ↓
┌────────────────────────────────────────────────┐
│  services/  (业务层)                            │
│   - llm/   : LLM 客户端封装 + LCEL chain        │
│   - rag/   : 文档处理 + 检索管道                │
│   - agent/ : LangGraph 状态机                   │
└────────────────────┬───────────────────────────┘
                     ↓
┌────────────────────────────────────────────────┐
│  schemas/  (数据契约)                           │
│   - Pydantic v2 模型，前后端共享语义             │
└────────────────────┬───────────────────────────┘
                     ↓
┌────────────────────────────────────────────────┐
│  core/  (基础设施)                              │
│   - config : 全局配置（pydantic-settings）       │
│   - logger : 日志（loguru，stdout 优先）        │
└────────────────────────────────────────────────┘
```

**为什么这么分？**

- **api 层薄**：只做参数校验和编排，几行就调 service
- **services 内聚**：每个子领域（llm / rag / agent）独立可测
- **schemas 统一**：所有跨层 DTO 走 Pydantic，IDE 全程补全

### 2.2 前端分层

```
src/
├── app/             # Next.js App Router 路由层
```

## 三、核心数据流

### 3.1 文档上传流程

```
[用户拖文件]
    │
    ↓ POST /api/v1/documents/upload
[FastAPI 接收 multipart]
    │
    ↓ ingest_document(file, doc_id, name)
[解析 (PyMuPDF / docx / md)]
    │
    ↓ raw text
[切片 (RecursiveCharacterTextSplitter)]
    │
    ↓ List[LCDocument]
[批量 Embedding (每批 10，并发)]
    │
    ↓ List[List[float]]
[写入 Milvus (HNSW 索引)]
    │
    ↓
[BM25 索引失效 → 下次检索时重建]
    │
    ↓
返回 {document_id, chunks_count}
```

**关键点：**
- DashScope text-embedding-v3 单批最多 10 条 → `_embed_in_batches` 用 `asyncio.gather` 并发跑批
- Milvus `client.insert` 是同步 → `asyncio.to_thread` 包一层避免阻塞事件循环
- BM25 是内存索引，依赖 Milvus 全量分块 → 入库后调 `invalidate()` 让下次检索重建

### 3.2 RAG 查询流程

```
[用户提问 + document_ids]
        │
        ↓
[Step 1] Query 改写（可选）
        │  use_multi_query → ["原问题", "变体1", "变体2"]
        │  use_hyde        → ["生成的假设答案"]
        ↓
[Step 2] 多 query 并行检索
        ├── 向量检索 (top_k=20)
        └── BM25 检索 (top_k=20)
        │
        ↓ RRF 融合 (k=60)
        │
        ↓ 候选 ~30 条
[Step 3] Reranker 精排 (gte-rerank-v2)
        │  query + 30 候选 → top_5 + score
        ↓
[Step 4] Prompt 拼装
        │  system: 强约束（只用提供的资料 / 找不到说不知道）
        │  human:  context={top_5}\n\n问题：{question}
        ↓
[Step 5] LLM 生成（流式）
        │
        ↓ SSE: {"type":"content", "data":"..."} × N
        ↓ SSE: {"type":"citations", "data":[...]}
        ↓ SSE: [DONE]
```

### 3.3 Agent 推理流程

```
[用户提问]
    │
    ↓
[StateGraph 启动]
    │
    ↓
┌──────────────────────────────────┐
│ agent_node (LLM 决策)            │
│  绑定 4 个工具 schema 给 LLM      │
│  LLM 输出 AIMessage              │
└──────────────────────────────────┘
    │
    ↓ has_tool_calls?
    │
   ┌┴┐
   │ │
 是 │ │ 否
   │ │
   ↓ ↓ END (返回最终回答)
┌─────────────────────────────────┐
│ TrackedToolNode                  │
│  · 顺序执行每个 tool_call        │
│  · 记录 (name, args, summary)    │
│  · 每个 tool 的结果作为           │
│    ToolMessage 加回 messages     │
└─────────────────────────────────┘
    │
    ↓ (回到 agent_node 继续推理)
    │
    ↓ MAX_ITERATIONS=10 防死循环
```

**关键设计：**
- `state.messages` 用 `Annotated[..., add_messages]` reducer，框架自动追加新消息
- `TrackedToolNode` 继承 `ToolNode`，在执行前后记录 metadata（用于前端 trace）
- `astream_events(version="v2")` 推送细粒度事件
- 关键过滤：只推 `content` chunk，跳过 `tool_call_chunks`（不然前端会看到 LLM 的"内部思考"）

## 四、关键模块依赖图

```
                     ┌───────────────┐
                     │   api/v1      │
                     └───────┬───────┘
              ┌──────────────┼──────────────┐
              ↓              ↓              ↓
       ┌──────────┐   ┌──────────┐   ┌──────────┐
       │ chat.py  │   │  rag.py  │   │ agent.py │
       └────┬─────┘   └────┬─────┘   └────┬─────┘
            │              │              │
            ↓              ↓              ↓
       ┌────────┐    ┌──────────┐  ┌─────────────┐
       │ llm/   │    │  rag/    │  │  agent/     │
       │ chains │←───│advanced_ │  │  graph      │
       │        │    │retrieval │  │   ↓         │
       └────────┘    └────┬─────┘  │  tools ─────┘
                          │        │   ↓
                ┌─────────┼────┐   │   ├─→ rag/retrieval
                ↓         ↓    ↓   │   ├─→ tavily
            vector_   bm25   re-   │   ├─→ calculator
             store         ranker  │   └─→ time
                                   └────────────┘
```

模块设计原则：

- **agent 调 rag**：Agent 把 RAG 当一个工具，没有循环依赖
- **rag 不调 agent**：RAG 是底层能力
- **llm/chains**：放最薄的 LCEL（prompt | model | parser），不耦合检索逻辑

## 五、状态管理

### 5.1 后端：无状态 + LangGraph 内存

- HTTP API 每次请求独立（包括 history 由前端带过来）
- LangGraph 用 `MemorySaver` 在单次请求内保存中间状态
- 跨请求会话：暂未实现（设计上预留 `thread_id` 支持后续接 PostgreSQL Checkpoint）

### 5.2 前端：Zustand store

```typescript
useChatStore: {
  messages: ChatMessage[],           // 对话历史
  isStreaming: boolean,              // 是否正在流式接收
  appendMessage(msg),                // 追加消息
  appendContentToLastAssistant(s),   // 流式追加 token
  addToolCallToLastAssistant(tc),    // Agent 工具调用入栈
  updateToolCallResult(id, result),  // 工具结果回填
  reset(),                           // 清空对话
}
```

为什么用 Zustand 不用 Redux：
- 体积小（~1 KB）
- 无 boilerplate，直接函数式 API
- TypeScript 友好

## 六、并发模型

### 6.1 异步优先

后端**几乎所有 IO 都是 async**：

| 调用 | 原生支持 | 处理 |
|------|---------|------|
| HTTP（FastAPI / httpx）| ✅ | 直接 await |
| LangChain LLM | ✅ | `chain.ainvoke / astream` |
| Embedding | ✅ | `aembed_documents` |
| Milvus | ❌（pymilvus 同步） | `asyncio.to_thread` 包装 |
| Tavily | ❌（SDK 同步） | `asyncio.to_thread` 包装 |
| 文件解析（PyMuPDF）| ❌（CPU 密集） | `asyncio.to_thread` |

### 6.2 关键并发点

```python
# 高级 RAG 检索同时跑两路
results = await asyncio.gather(
    vector_search(query, top_k=20),
    bm25.search(query, top_k=20),
)

# Multi-Query 时多 query 并行
per_query_results = await asyncio.gather(*[
    _retrieve_for_single_query(q, cfg, doc_ids) for q in queries
])

# Embedding 分批并发
batch_results = await asyncio.gather(*[
    embedding_model.aembed_documents(b) for b in batches
])
```

## 七、错误处理与降级

| 故障点 | 处理策略 |
|--------|---------|
| Milvus 不可达 | 启动时连不上 → 抛错；运行时 → 5xx + 日志 |
| LLM 限流 / 超时 | tenacity 重试 3 次（指数退避） |
| Reranker 调用失败 | 降级：保持原序，统一打 0.5 分（不阻断流程） |
| Tavily Key 缺失 | 优雅降级：返回 mock 文案，提示用户配 key |
| 文档解析失败 | 立刻返回 4xx，不入库 |
| BM25 索引为空 | 降级为仅向量检索 |

## 八、安全与隔离

- 容器内非 root 用户（`app` / `nextjs`）
- DashScope / Tavily Key 通过环境变量注入，不进镜像
- CORS 白名单（默认 localhost:3300）
- 上传大小限制 50 MB
- Calculator 工具用沙箱 eval，禁止 import / 内置函数

## 九、可观测性（现状 + 规划）

### 现状
- **日志**：loguru，stdout + 文件双轨；容器环境优雅降级到仅 stdout
- **健康检查**：`/api/v1/health`，Docker Compose `healthcheck` 联动
- **测试脚本**：`scripts/test_rag.py` / `test_agent.py` 端到端验证

### 规划
- **链路追踪**：接 Langfuse 看每条对话的 LLM 调用 / token 用量 / 延迟
- **Metrics**：Prometheus 暴露 QPS / latency / error rate
- **告警**：成本超额 / 错误率突增 → 邮件/钉钉
